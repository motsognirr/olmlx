# olmlx

Ollama-compatible API server using Apple MLX for local inference on Apple Silicon.

## Usage Context

Single-user, localhost-only inference server:
- No auth, rate limiting, or TLS — localhost-bound by default
- Error messages may include internal details (paths, model names)
- The inference lock provides natural backpressure

## Non-Obvious Invariants

**Metal stream hazard** — All non-speculative inference (prefill + decode) must run on `generation_stream`. Mixing streams leaves GatedDeltaNet/MoE recurrent state as a cross-stream lazy graph that corrupts expert routing on Qwen3.x hybrids — output looks coherent but is pretraining-data dumps. Speculative decoders are the exception: both prefill and decode run on `default_stream` (a split would reintroduce the same hazard). `snapshot_cache_for_persistence` does deepcopy + eager `mx.eval` before snapshots cross worker threads — lazy graphs bound to one stream crash on reuse from another.

Since mlx 0.32.0 (#499) streams are **thread-local**: `generation_stream` is a per-thread proxy, so "same stream" is enforced as "same thread" — the checkpoint-prefill drive is deferred into the generation worker thread (`deferred_prefill` closure), never run on the event loop. Worker GPU work is fenced by the worker's own end-of-run sync + thread join; the event-loop thread never syncs generation streams. `PromptCacheStore` stays loop-affine — worker-side checkpoint inserts marshal through `loop.call_soon_threadsafe`.

**Pure-RotatingKVCache prefill is single-chunk** — Models with only SWA (`RotatingKVCache`, no `ArraysCache`) — gpt-oss, Step-3.5, Gemma 3 — must prefill in ONE `model()` call. Splitting at a message boundary corrupts windowed attention: output is coherent but unrelated, and tool calls are silently skipped (tool defs add the system segment that creates the boundary). Detected by `_is_pure_rotating_cache`.

**Grammar tokenizer identity** — CPython recycles `id()` addresses; each grammar cache entry carries a `weakref` to its tokenizer and validates referent identity before serving. A stale entry with a recycled id would serve a wrong-vocabulary grammar. `drop_for_tokenizer` in `_close_loaded_model` runs **first** so no other close-step failure can skip it.

**TurboQuant deepcopy** — `TurboQuantKVCache.__deepcopy__` shares `mx.Dtype` by reference (pickle rejects it) and eager-evals private dequant side buffers — they're not exposed by `flatten_cache_state` and would leave a Metal-stream-bound lazy graph that crashes on cross-thread reuse.

**TurboQuant dequant-buffer shed-on-store** — `TurboQuantKVCache` holds a full-precision dequant side buffer (`_key_dequant`/`_value_dequant`, ~4–8x the packed footprint at 4-bit). `_estimate_state_bytes` walks `__dict__` and counts it, so an unshed cache trips the `prompt_cache_ram_budget_gb` soft-eviction at only ~30k tokens — and with disk spill off (default) the entry is silently dropped, forcing a full re-prefill every turn after. `_shed_transient_buffers` (store.py) calls `release_dequant_buffers()` from `_set_in_memory` — the single insertion chokepoint for **every** store path (`async_set`/`insert_checkpoint`/disk-restore/`takeover`/preflight), so the shed can't be missed at an individual call site. `update_and_fetch` rebuilds the buffers lazily from packed indices+norms on resume (so the rebuild must run *before* the resize/splice paths), and `trim`'s buffer-shrink branch guards against the shed (`None`) buffers. Only TurboQuant is affected — Spectral dequantizes to locals, Shard dequantizes on read.

**Rotation matrices must be eager-eval'd at construction** — `TurboQuantRotation.__init__` and `SpectralRotation.__init__` precompute `matrix_T`/`V_T` as `matrix.T` — a *lazy* transpose op. The prompt cache (and its rotations) is built on the **event-loop thread** (`_make_prompt_cache_for_lm`), while prefill/decode run on a separate `asyncio.to_thread` worker. Under mlx ≥0.31.2 thread-local streams (#499) the lazy op stays bound to the constructing thread's stream, so evaluating any graph that references it from the worker thread raises `There is no Stream(gpu, N) in current thread` — surfacing at flash-MoE's `mx.eval(inds)` on the first token (bricked every `kv_cache_quant`-configured flash-MoE model). Both `__init__`s call `mx.eval(matrix, matrix_T)` so the rotations are materialized leaves, safe to read from any thread (materialized arrays carry no stream binding — same rationale as `_pin_state_to_offset`). The **TurboQuant codebook** (`get_codebook`) is the same trap on a different axis: it caches `centroids / mx.sqrt(dim)` — a *lazy* divide — in the **module-global** `_codebook_cache`, so the first thread to populate an entry binds it, and a *second concurrent request* whose worker thread references the shared codebook raises `There is no Stream(cpu, N) in current thread` (cpu, not gpu — the scalar sqrt/divide schedules on the CPU stream) at the same `mx.eval(inds)` site. Single requests dodge it (build + consume on one worker thread); it needs two in flight. Fix: `get_codebook` calls `mx.eval(codebook)` before caching. Not caught by the rotation test — that test happens to build the codebook on the thread that evaluates it.

**Flat-path cache reuse trims quant caches on the worker thread** — a *second* `no Stream(gpu, N)` crash (same `mx.eval(inds)` site) hit pure-MoE + `kv_cache_quant` models (GLM-4.5-Air, Qwen3-235B) once the rotation fix let them reach later requests. These set `uses_checkpoint_persistence=False` (TurboQuant rejects state restoration), so cross-request KV reuse goes through the **flat** path (`_setup_prompt_cache`), which calls `trim_prompt_cache` on the **event-loop thread**. A large trim takes `TurboQuantKVCache.trim`'s buffer-shrink branch (`_key_indices = _key_indices[..., :cap, :]` — a lazy slice) and the quant `.state` rebuilds a lazy `[:offset]` slice per access — both bound to the loop thread, crashing when the worker evaluates them. Fix: the flat path **defers the trim into the worker** via the existing `deferred_prefill` hook when `_cache_list_contains_lazy_state(working_cache)` (now flags `TurboQuant`/`Spectral`/`Shard` KVCaches by class name — *not* `hasattr(_pin_state_to_offset)`, which a MagicMock would spuriously match). A trimmable cache trims exactly `trim_amount` when `trim_amount ≤ offset` (always true here), so the deferred count is predicted rather than inspected. Hybrid-GDN models dodge this entirely — they use the thread-safe checkpoint path.

**TurboQuant flat-path packed buffers stay lazy-bound to the generating worker** — a *third* `no Stream(gpu, N)` crash (same `mx.eval(inds)` flash-MoE site) on GLM-4.5-Air / Qwen3-235B, surviving all of #653/#654/#655/#657: the **packed KV buffers themselves** (`_key_indices`/`_key_norms`/`_value_indices`/`_value_norms`) are never materialized on the flat store path. `update_and_fetch` returns views over the `_key_dequant`/`_value_dequant` **side** buffer, so the packed writes (`_key_indices[..., prev:offset, :] = k_idx`) never enter the returned token's graph — mlx-lm's per-token `mx.eval` leaves them a lazy `slice_update` chain bound to the generating worker thread's Metal stream. The **checkpoint** path materializes them via `snapshot_cache_for_persistence` → `_pin_state_to_offset` (`mx.eval` on a snapshot); the **flat** path (all trimmable TurboQuant models, `uses_checkpoint_persistence=False`) stores the *live* object with no such eval, so the next request's worker thread crashes rebuilding dequant / deferred-trimming from a dead-thread-bound graph. Only TurboQuant is hit — Spectral/Shard dequantize *from* the packed buffers on every fetch (so the token eval materializes them); TurboQuant's side-buffer design is exactly why its packed state stays write-only. Fix: `TurboQuantKVCache.ensure_state_materialized()` (`mx.eval`s the four packed buffers, no slice — the live cache keeps its `step`-aligned headroom) is called on the generating worker at end-of-generation via a duck-typed `materialize_lazy_cache_state` (streaming.py): the streaming path wraps the mlx generator in a `finally` finalizer (runs on `gen.close()`/exhaustion, before `_SENTINEL`), the non-streaming path calls it in `_generate_sync` before its `mx.synchronize`. Both run on the worker before the cache can cross into the store/next-request thread. Per-token eval was rejected (reintroduces the O(n²) materialization the side buffer exists to avoid). Not caught by #654's deferred-trim test (that mocked the wiring, never exercised real cross-thread lazy state). **Second re-laziness site:** the store-time `prompt_cache_max_tokens` trim in `_store_prompt_cache_after_generation` runs on the **event-loop thread** and, for a large shrink (`offset < capacity//2`, needs a >~64k-token conversation), takes `trim`'s buffer-shrink branch — re-binding the packed buffers to the *loop* thread's stream *after* the worker-side materialize already ran; `_shed_transient_buffers` only nulls the dequant side buffers, so that lazy slice would be stored as-is. Fixed by calling `materialize_lazy_cache_state(prompt_cache)` again right after that trim — on the loop thread that owns the fresh slice op (correct: the op is loop-bound, so loop-side eval materializes it). The eval is shared via `TurboQuantKVCache._eval_packed_buffers` (used by both `ensure_state_materialized` and `_pin_state_to_offset`).

**Flash-MoE wrapper must eval the INNER model's params, not the wrapper's** — `wrap_flash_moe` loads the base model with `lazy=True` (so routed experts are never materialized) and is expected to materialize the *non-expert* in-RAM weights before serving. `FlashMoeModelWrapper` stores the base under `self._model`, and mlx's `nn.Module.parameters()` does **not** traverse underscore-prefixed attributes — so `wrapped.parameters()` is **empty** and `mx.eval(wrapped.parameters())` materialized *nothing*, leaving every weight (embeddings, attention, router gates, norms) lazy and bound to the load thread. Under mlx thread-local streams (#499) the model is loaded on one `asyncio.to_thread` worker and generation runs on a *different* one, so the forward crashed evaluating load-thread-bound lazy weights — `There is no Stream(cpu, N) in current thread` at flash-MoE's `mx.eval(inds)`, with `embed_tokens.weight` the first lazy leaf hit. Deterministic (not a race), single-request, hit every `kv_cache_quant` flash-MoE model (GLM-4.5-Air, Qwen3-235B). Distinct from the rotation (#653) and flat-path trim (#654) crashes — those are per-request cache state; this is the model weights themselves. Fix: `wrap_flash_moe` evals `model.parameters()` (the local inner model, mutated in place by `_replace_moe_layers` so routed experts are already gone from the tree — exactly the non-expert set), *not* `wrapped.parameters()`. Was masked in single-threaded harnesses (bench) where load + forward share a thread, so the lazy weights bind to and eval on the same thread.

**Scaled-RoPE `_freqs` must be materialized at load** — the *same* `nn.Module.parameters()` underscore-traversal gap as the flash-MoE wrapper crash, but on a plain mlx-lm text load. mlx-lm's `load` does `mx.eval(model.parameters())`, but the scaled-RoPE variants (`YarnRoPE`/`SuScaledRoPE`/`Llama3RoPE`/`ProportionalRoPE` in `mlx_lm/models/rope_utils.py`) precompute `self._freqs` (and `_scale`) as a *lazy* array in `__init__` under an **underscore key**, so `parameters()` skips it and the load-time eval never materializes it — it stays bound to the load thread's default stream. Under #499 thread-local streams the model loads on one worker and generates on another, so the first forward that evals a graph referencing `_freqs` raises `There is no Stream(gpu, 0) in current thread` — deterministic, every request. Surfaced on the 1M-context Qwen3.5 hybrid `empero-ai/Qwythos-9B` (`rope_type: "yarn"`); the crash lands at the deferred-prefill `mx.eval(flatten_cache_state(cache))` (inference.py `_drive_segmented_prefill._run`) because that's the first cross-thread eval, but the lazy leaf is the RoPE buffer, not the cache. Models on default/mrope RoPE carry no `_freqs` and are unaffected (Qwen3-32B-4bit opencode worked). Fix: `_materialize_module_buffers(model)` (model_manager.py) walks `named_modules()` and eager-evals every underscore-keyed array leaf (`nn.Module` is a `dict`, so buffers live in `.items()`, not `__dict__`) on the load thread. Materialized leaves carry no stream binding, so they read from any worker thread. It only touches small `__init__`-computed constants (never the possibly-`lazy=True` weights — those are non-underscore parameters), so it's safe on the flash/flash-MoE paths that deliberately keep weights lazy. Wired at **all three loader chokepoints**: `_load_with_model_type_fallback` (mlx-lm text + the flash-MoE mlx-lm branch, which routes through it), `load_vlm` (the single mlx-vlm chokepoint — main VLM, lm-then-vlm fallback, and both flash / flash-MoE VLM fallbacks), and `load_model_with_strict_fallback` (flash dense/sparse-FFN). `load_vlm`/`prepare.py` lazy-import the helper from model_manager to avoid an import cycle. Gated by `tests/test_thread_local_streams.py::TestScaledRopeBufferMaterialization` (core mechanism, with a negative control that locks the crash in) and `::TestScaledRopeBufferMaterializationWiring` (each chokepoint, via monkeypatched loaders).

**Whisper/TTS loaders must materialize buffers too** — the *same* underscore-buffer gap as scaled-RoPE `_freqs`, but on the **whisper** and **TTS** loader chokepoints, which don't route through the mlx-lm/mlx-vlm/flash paths that already call `_materialize_module_buffers`. `mlx_whisper.load_models.load_model` runs `mx.eval(model.parameters())`, but `mlx_whisper.whisper.AudioEncoder._positional_embedding` (`sinusoids(...).astype(dtype)`) and `TextDecoder._mask` (`create_additive_causal_mask(...).astype(dtype)`) are lazy arrays under underscore keys, so they stay bound to the **load** thread's stream. Whisper *load* and *transcribe* are separate `asyncio.to_thread` calls: on a fresh server they usually land on the same pool thread (crash masked), but once prior traffic has rotated the executor they land on different threads and the first transcribe forward aborts the process with the uncatchable `libc++abi ... There is no Stream(gpu, N) in current thread` (#651) — **no client disconnect required**; the disconnect in the original report was a red herring (it just churned pool threads). This is *not* a gc/reference-cycle bug (the first-comment hypothesis), and not fixed by any `mx.synchronize`. Fix: the whisper branch is extracted to `ModelManager._load_model_whisper` (mirroring `_load_model_tts`) and both call `_materialize_module_buffers(model)` after loading. Kokoro carries no lazy underscore buffers, but the TTS loader is generic over all mlx-audio models (fish/spark/indextts stash lazy `_cos`/`_sin`/`_pe`/`_inv_freq`), so it gets the same treatment defensively. The reranker (XLM-RoBERTa) uses proper `nn.Embedding` params — no lazy underscore buffers, audited and left unchanged. Gated by `tests/test_thread_local_streams.py::TestWhisperBufferMaterialization` (mechanism + negative control on whisper's real `sinusoids`/`_mask` buffers) and `::TestWhisperTtsLoaderWiring` (both loaders, via monkeypatched loaders); verified end-to-end against real `whisper-tiny` (raw load → cross-thread forward crashes; `_load_model_whisper` → forward succeeds).

**Hybrid VLM routing** — VLMs whose `text_config.layer_types` contains `"linear_attention"` (Qwen3.5, Qwen3_5_moe) route through mlx-lm, not mlx-vlm. The mlx-vlm path crashes with a Metal stream error on text inference.

**Distributed: eval before first forward** — After `model.shard()`, must `mx.eval(model.parameters())` on every rank before any forward pass. The combined lazy materialization + `all_sum` Metal command buffer times out (~10s GPU timeout) on 32B+ models.

**Distributed: signal handlers before spawn** — Worker cleanup signal handlers must be installed before spawning SSH workers. The default SIGTERM disposition skips atexit, which orphans SSH workers if the coordinator dies during the pre-uvicorn startup window.

**MTP concat order** — In `MTPDecoder`, the fc input is `concat([embed, hidden])` — embed first, which is the **opposite** of EAGLE's order. Getting it wrong drops acceptance to ~0.006 while remaining exactness-preserving (correct but useless). Determined empirically via `scripts/mtp_decoder_probe.py`.

**MTP prefill chunking** — `MTPDecoder.prefill` pass-1 must chunk through `_chunked_prefill`. A single forward over a long prompt forces `lm_head` over every position, producing a `[1, seq-1, vocab]` float32 tensor that OOMs Metal (~44 GB at 73k tokens vs ~41.7 GB limit). GDN capture is suppressed (`use_buffer(None)`) during the chunked prefix — the capture call appends per forward, so leaving it active rebloats memory across sub-chunks on hybrid GDN targets.

**DFlash training** — Always use `--self-generate`. Ground-truth dataset text → ~0.4% acceptance. Target-generated responses → real acceptance (~45% p1 on Qwen3-0.6B). This is the upstream recipe.

**TTS priming** — The router primes the PCM source generator's first chunk before starting the response. Priming the ffmpeg-encoded stream instead wouldn't work: ffmpeg emits its container header before reading stdin, so an upstream `ValueError` from a non-TTS model would leak after the 200 has been sent.

**Flash prefetcher teardown order** — Prefetcher must be closed before the weight store on model unload. Prefetch tasks submit work into the store's thread pool; reversing the order leaves in-flight tasks referencing a closed store.

**Flash-MoE router dispatch order** — `_replace_moe_layers` picks a replacement class by gate shape, and the order of the `elif` chain is load-bearing. A plain `nn.Linear`/`QuantizedLinear` gate returns logits; a custom gate (DeepSeek/Kimi) returns `(inds, scores)` directly. The two linear-gate special cases (Qwen3-Next = `shared_expert_gate`, MiniMax = `e_score_correction_bias`) and the plain-Qwen3Moe case (`_FlashMoEQwen3`, e.g. Qwen3-235B-A22B) must all be matched **before** the `gate is not None` DeepSeek branch — otherwise a linear gate falls into `_FlashMoEDeepSeek._route`, which unpacks `inds, scores = self.gate(x)` and dies with `not enough values to unpack (expected 2, got 1)` on the first forward.

**Flash-MoE expert parse skips a defensive copy** — an expert-cache miss's cost is ~75% parse (host-side `mx.array` construction), not I/O: experts are page-cache-resident, so `pread` runs at memcpy speed (~157 µs for a 1.69 MiB expert). The three `_parse_*` methods (`moe_weight_store.py`) therefore parse from a **zero-copy `memoryview`** and let `mx.array` do the *one* necessary host→device copy — no bytes-slice, no `np.frombuffer(...).copy()`. This is safe because `mx.array` copies host data at construction (so parsed arrays never alias `raw`, even though `raw` is freed before the deferred eval); `test_parse_does_not_alias_source_buffer` locks that invariant. Removing the redundant copy was worth **+7–23% decode throughput** (bigger at low cache budgets, where misses dominate). Corollary — the reason predictive expert prefetch/lookahead (#650) never beat LRU and was **removed** (2026-07-11): I/O was never the bottleneck (experts are page-cache-resident), so speculative reads only added CPU/copy contention, and the router is load-balanced (near-uniform) so no causal predictor can anticipate reuse — the Flash-MoE store is now plain `LayerLruCache`, no `ScoredLayerCache`, no `MoePrefetcher`/`MoeLookaheadBank`. The layout offset-table `.copy()` at load time is unrelated and correct (the backing `f.read()` bytes go out of scope).

**Autonomous agent rides ChatSession** — `engine/agent/` (gated on `OLMLX_AGENT_ENABLED`) drives goal-pursuit by wrapping the existing `ChatSession` ReAct loop, never calling MLX directly — so the inference-lock / Metal-stream handling is reused unchanged (same constraint as the panel coordinator). The one behavioral extension is *continuation semantics*: a stop without the `finish` tool injects a nudge and the run continues. `finish` is detected from the `tool_call` **event stream**, not a return value; `AgentToolManager` subclasses `BuiltinToolManager` so `ChatSession` dispatches the control tools (`finish`/`remember`/`recall`/`create_skill`/`delegate`) transparently. `AgentStore` shares one SQLite connection across `asyncio.to_thread` worker threads (`check_same_thread=False` + a `threading.Lock`); all runtime I/O is offloaded so the event loop never blocks on disk. Subagents share the global inference lock — `DelegateRunner` serializes child generation behind a semaphore (a concurrency-limited queue, not wall-clock parallelism). **Tool safety (#611):** the headless agent has no human to CONFIRM and can ingest untrusted web content via `web_fetch`, so `_default_session` installs a `ToolSafetyPolicy` (default `ALLOW`, so safe tools still run autonomously) that gates the mutating/exec builtins — `bash` via `agent_shell_policy`, `write_file`/`edit_file` via `agent_file_write_policy` (each `allow`/`auto`/`deny`, default `auto`). `auto` routes each call through `_make_tool_safety_judge` (an LLM verdict via `generate_chat`, **fail-closed**: only an explicit `ALLOW`-without-`DENY` passes). This requires `local_tool_safety=True` because the dangerous builtins are *local* tools, which otherwise bypass the policy (`_classify_tool_calls`). File writes are additionally sandboxed to `ChatConfig.write_root` (`agent_workspace_dir`, default cwd) — `_resolve_path(confine_root=...)` now rejects absolute-path escapes, closing the gap where its traversal guard only covered relative paths. The single `_make_session` chokepoint means delegated subagents inherit the same policy. The judge shares the agent model, so it is defense-in-depth, not a guarantee against a fully compromised model — `deny` is the hard-off posture.

**`/api/generate` context continuation feeds token ids, not a string** — the Ollama legacy `context` field (#656) makes `generate_completion` tokenize the prompt, prepend any prior `context` tokens (`build_context_input_tokens` drops a doubled leading `bos_token_id`), and pass the resulting **`list[int]`** as `prompt` so the returned `context` is a faithful continuation prefix; the response/terminal-chunk `context` = those input tokens + the generated token ids. Generated ids are surfaced via `collect_generated_tokens` (threaded to `_full_completion`/`_stream_completion`); a context request also **bypasses the batched path** (`_full_completion_batched` aggregates and can't emit per-token ids). Text models only — VLM token ids can't represent image patches, so the VLM path skips it. Only `/api/generate` opts in (`return_context=True`); `/v1/completions` (the other `generate_completion` caller) is untouched.

**Panel coordinator routes through generate_chat** — `engine/panel.py` is a per-turn reconciler that rides the client's tool loop: it returns a `generate_chat`-shaped result whose text is either the judge's prose or canonical Qwen `<tool_call>` blocks, so both routers' existing `parse_model_output` paths handle it unchanged. Every classifier/panelist/judge call MUST go through `generate_chat` — never call MLX directly — or the Metal-stream/inference-lock handling is bypassed. Panels need `max_loaded_models ≥ members + judge + classifier` to avoid reload thrash.

**Proxy-tuning is a non-exactness-preserving speculative strategy** — `engine/proxy_tuning.py`'s `ProxyTuningDecoder` registers as `speculative_strategy="proxy_tuning"` to reuse the speculative lifecycle/stream/dispatch plumbing, but it *alters* the output distribution (`base + α·(expert − antiexpert)`), so it deliberately has no draft→verify→accept and no cache trimming — every model advances over the same committed tokens. Expert/anti-expert are loaded **inline by the loader** (held by the decoder, not in `ModelManager._loaded`), so no `max_loaded_models` bump is needed — unlike the panel coordinator. All three models must share one exact tokenizer/vocabulary (`check_vocab_identity` enforces token-mapping equality, stricter than `_check_vocab_match`'s size-only test). Supports **dense and hybrid GDN/MoE** bases: it installs no GDN capture because that only undoes rejected draft tokens and proxy-tuning never speculates (no draft→verify→accept, no trimming), so per-token forward + `make_prompt_cache` keeps GDN recurrent state correct on the default stream; verified on a hybrid Qwen3.5 base (`tests/test_proxy_tuning_hybrid.py`). Grammar disables it per-request like every other speculative strategy.

**LoRA-adapter hot-swap shares base weights via a structural copy** — `base:adapter` entries (`engine/registry.py` `AdapterConfig`, `"adapters"` section of models.json) load through `ModelManager._load_adapter_model`, which `structural_copy`s the base model and applies `mlx_lm.tuner.utils.load_adapters`. The copy is deliberately neither `copy.copy` (shallow — shares the `layers` list/submodules, so `load_adapters` would corrupt the base) nor `copy.deepcopy` (duplicates `mx.array` weights, and crashes on quantized models because `mx.Dtype` is unpicklable): it rebuilds the `Module`/list/dict tree with `mx.array` leaves and scalar attrs (`bits`, dtypes) shared by reference, and copies the per-module `_no_grad` set so freezing one instance can't affect another. The adapter `LoadedModel` reuses the **same tokenizer object** as the base, so `_close_loaded_model` skips `drop_for_tokenizer` for adapter entries (`lm.adapter_base` set) — the base drops it once its last adapter is gone. The base is pinned against LRU/expiry eviction while `_adapter_child_refs > 0` (incremented under the manager lock at load, decremented via `_detach_adapter_locked` at every `_loaded` removal); unloading a base with live adapters raises `ActiveRequestsError`. Needs `max_loaded_models ≥ base + adapters` (each occupies a slot). VLM/flash/distributed/speculative/KV-quant/whisper/tts/reranker bases are rejected (`_reject_adapter_base`). Verified end-to-end on a real model in `tests/test_adapter_loading.py` (Metal-gated).

## Type Annotations

- **TypedDict**: `total=True` for required keys, `total=False` for optional. Never empty — use `dict[str, Any]` for freeform JSON.
- **Pydantic field types**: Prefer `dict[str, Any]` over TypedDict for free-shape fields (e.g. `Tool.function`) — Pydantic v2 enforces required keys and silently strips extras.
- **Protocol return types**: `__aexit__` returns `bool | None` (not `None`) to allow suppression. `__call__` must match actual return type (mlx-lm models return `mx.array`, not `dict`).
- **AsyncIterator**: If yielding both `str` and `dict`, annotate `AsyncIterator[str | dict[str, Any]]`.
- **TYPE_CHECKING**: Guard annotation-only imports when module has `from __future__ import annotations`.
- **Protocol return values**: Match the real type (e.g. `ClientSession.initialize()` returns `InitializeResult`).
